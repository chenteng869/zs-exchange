### 2.3 推荐：混合架构（自研 + Fireblocks + BitGo）

```
用户充提 ←→ 自研账本层 ←→ 热钱包（Fireblocks MPC）
                                      ↓
                              温钱包（Fireblocks 多签）
                                      ↓
                              冷钱包（BitGo 硬件托管）
```

**核心逻辑：**
- **热钱包**：Fireblocks API 驱动，自动处理用户充提，限额自动触发审批
- **温钱包**：Fireblocks Policy Engine，人工审批 + 定时调拨
- **冷钱包**：BitGo 保险托管，月度审计，离线签名
- **自研层**：账户系统、账本系统、风控系统、对账系统全部自研

### 2.5 钱包安全架构要点

#### 2.5.1 私钥管理
- **MPC 门限签名**：2-of-3 / 3-of-5 门限，私钥永不完整存在
- **HSM 硬件安全模块**：冷私钥存储在 FIPS 140-2 Level 3 HSM 中
- **密钥分片**：私钥分片存储在不同地理位置
- **密钥轮换**：定期轮换密钥，事件触发即时轮换

#### 2.5.2 审批策略
```
提现金额 < $10K     → 自动通过
$10K < 金额 < $100K → 运营 + 风控 双审批
$100K < 金额 < $1M  → 运营 + 风控 + 财务 三审批
金额 > $1M          → COO + CFO 双签 + 冷钱包手动操作
```

#### 2.5.3 地址风控（前置拦截）
| 风险等级 | 判定标准 | 处置措施 |
|---------|---------|---------|
| 高风险 | 制裁名单、暗网、混币器 | 拦截 + 上报合规 |
| 中风险 | 可疑地址、风险评分 > 70 | 人工审核 + 延迟出金 |
| 低风险 | 已知交易所、合规机构 | 正常处理 |

### 3.5 零知识证明（ZK）在 DID 中的应用

**核心场景：选择性披露**

```
用户需要证明："我已通过 KYC，且年满 18 岁"
传统方式：暴露全名、身份证号、生日 → 隐私泄露
ZK 方式：生成零知识证明 → 只暴露"验证通过"结论 → 隐私保护
```

**ZK 用例规划：**
1. **年龄证明**：证明年满 18/21 岁，不暴露具体生日
2. **国籍证明**：证明来自合规国家，不暴露具体国籍
3. **余额证明**：证明资产 > $X，不暴露具体金额
4. **KYC 状态证明**：证明已通过 KYC，不暴露 KYC 细节
5. **机构资质证明**：证明是合规机构，不暴露机构名称
