## 1. 概述

### 1.1 目的

本文档定义了中萨数字科技交易所管理员后台的完整运作机制，涵盖组织架构、权限管理、审批流程、审计追踪、安全防护、操作规范等核心内容，确保管理员后台安全、高效、合规地运行。

### 1.2 适用范围

- 超级管理员
- 平台管理员
- 各业务模块管理员
- 财务审核人员
- 风控合规人员
- 客服运营人员
- 技术运维人员

### 1.3 核心原则

| 原则 | 说明 |
|------|------|
| **最小权限原则** | 管理员仅拥有完成其工作职责所需的最少权限 |
| **职责分离原则** | 关键操作需多人参与，避免单一管理员拥有过大权力 |
| **全程审计原则** | 所有管理员操作必须记录在案，可追溯、可审计 |
| **双人复核原则** | 高风险操作必须经过第二人复核确认 |
| **定期轮换原则** | 管理员权限定期审计，关键岗位定期轮换 |

### 2.2 角色定义

#### 2.2.1 超级管理员 (Super Admin)

**人数**：2人（双控机制，需同时授权）

**职责**：
- 系统最高权限，负责全局配置
- 管理员账号的创建与注销
- 角色权限体系的调整
- 重大操作的最终审批
- 应急情况下的特权操作

**权限范围**：全部权限（但高风险操作需双签）

**人数**：1人

**职责**：
- 平台日常运营管理

- 运营团队管理
- 跨部门协调

**权限范围**：
- 用户管理（查看/编辑/冻结）
- 内容管理（发布/审核/下架）

## 3. RBAC权限管理体系

### 3.1 权限模型

采用 **RBAC (Role-Based Access Control)** 模型，基于角色进行权限管理：

```
用户 → 角色分配 → 角色 → 权限关联 → 权限点
```

### 3.2 权限分层结构

```
系统级权限
├── 用户管理 (users)
│   ├── users:view          # 查看用户
│   ├── users:create        # 创建用户
│   ├── users:edit          # 编辑用户
│   ├── users:delete        # 删除用户
│   ├── users:freeze        # 冻结用户
│   └── users:kyc           # KYC审核
├── 交易管理 (transactions)
│   ├── transactions:view   # 查看交易
│   ├── transactions:deposit # 充值审核
│   ├── transactions:withdraw # 提现审核
│   └── transactions:freeze # 冻结交易
├── 内容管理 (content)
│   ├── content:view        # 查看内容
│   ├── content:create      # 创建内容
│   ├── content:edit        # 编辑内容
│   ├── content:delete      # 删除内容
│   └── content:audit       # 内容审核
├── 财务管理 (finance)
│   ├── finance:view        # 查看财务
│   ├── finance:deposit     # 充值审核
│   ├── finance:withdraw    # 提现审核
│   ├── finance:settlement  # 结算管理
│   └── finance:reconciliation # 对账管理
├── 系统管理 (system)
│   ├── system:config       # 系统配置
│   ├── system:roles        # 角色管理
│   ├── system:admins       # 管理员管理
│   ├── system:audit        # 审计日志
│   └── system:server       # 服务器监控
└── 风控管理 (risk)
    ├── risk:view           # 查看风控
    ├── risk:rules          # 风控规则
    ├── risk:events         # 风险事件
    └── risk:freeze         # 冻结账户
```

### 3.3 权限管理流程

#### 3.3.1 权限申请流程

```
申请人提交申请 → 直属上级审批 → 部门负责人审批 → 风控合规审核 → 超级管理员授权 → 权限生效
```

#### 3.3.2 权限变更流程

```
变更发起 → 说明变更原因 → 直属上级审批 → 风控合规审核 → 超级管理员审批 → 执行变更 → 通知相关人员
```

#### 3.3.3 权限回收流程

```
触发条件（离职/转岗/违规）→ HR或部门发起 → 直属上级确认 → 风控合规审核 → 立即回收权限 → 审计记录
```

### 3.4 权限定期审计

| 审计类型 | 频率 | 负责人 | 内容 |
|---------|------|--------|------|
| 权限清单审计 | 每月 | 风控合规 | 所有管理员权限清单核对 |
| 操作行为审计 | 每周 | 风控合规 | 高权限账号操作行为分析 |
| 权限合理性审计 | 每季度 | 风控+运维 | 权限与职责匹配度检查 |
| 全面审计 | 每半年 | 外部审计 | 全面权限体系审计 |

## 4. 工作流审批机制

### 4.1 审批级别定义

| 级别 | 定义 | 适用场景 | 审批人 |
|------|------|----------|--------|
| L1 | 单人审批 | 常规操作、低风险 | 直属上级 |
| L2 | 双人审批 | 中等风险操作 | 部门负责人 + 风控 |
| L3 | 三人审批 | 高风险操作 | 总监 + 风控 + 超级管理员 |
| L4 | 双签审批 | 极高风险操作 | 2名超级管理员同时确认 |

### 4.2 关键操作审批流程

#### 4.2.1 用户冻结/解冻

**风险等级**：L2（双人审批）

**流程**：
```
运营管理员发起 → 填写冻结原因/证据 → 运营总监审批 → 风控合规审核 → 执行冻结/解冻
                                                                ↓
                                                          通知用户（站内信/邮件）
```

**时间要求**：
- 紧急冻结：30分钟内完成审批
- 常规冻结：24小时内完成审批

---

#### 4.2.2 大额提现审核

**风险等级**：L2（双人审批）- L3（三人审批，视金额而定）

**触发阈值**：
- 10,000 USDT 以下：财务单人审核（L1）
- 10,000 - 100,000 USDT：财务 + 风控审核（L2）
- 100,000 USDT 以上：财务 + 风控 + 总监（L3）

**流程**：
```
系统自动风控检测 → 财务初审（核对信息/人工复核）
                    ↓
              风控审核（反洗钱/可疑分析）
                    ↓
              总监审批（超阈值时）
                    ↓
              执行出款 → 通知用户
```

---

#### 4.2.3 管理员权限变更

**风险等级**：L3（三人审批）

**流程**：
```
申请人提交 → 直属上级审批 → 部门负责人审批 → 风控合规审核 → 超级管理员授权 → 生效
```

---

#### 4.2.4 系统配置变更

**风险等级**：L2 - L3

**流程**：
```
运维工程师提交变更方案 → 技术总监审批 → 风控合规审核 → 执行变更（窗口期）→ 验证结果 → 通知相关方
```

**变更窗口期**：
- 常规变更：每周二、四 02:00-04:00（低峰期）
- 紧急变更：随时执行，但需事后补审批流程

---

#### 4.2.5 上币/下币操作

**风险等级**：L3（三人审批）

**流程**：
```
项目方提交申请 → 项目审核（技术/合规/市场）→ 上币委员会评审 → 运营总监审批
                                                                 ↓
                                                          风控合规审核
                                                                 ↓
                                                          超级管理员批准
                                                                 ↓
                                                          执行上币/下币
```

---

### 4.3 审批超时处理

| 审批级别 | 超时时间 | 处理方式 |
|---------|----------|----------|
| L1 | 2小时 | 自动升级至上级 |
| L2 | 4小时 | 自动通知部门总监 |
| L3 | 8小时 | 自动通知所有超级管理员 |
| L4 | 24小时 | 触发应急流程 |

## 5. 审计日志系统

### 5.1 日志分类

| 日志类型 | 记录内容 | 保存周期 |

| user_agent | 浏览器UA | Mozilla/5.0 ... |
| request_id | 请求ID | req-abc123def |
| before_data | 变更前数据 | {status: 'active'} |
| after_data | 变更后数据 | {status: 'frozen'} |
| details | 操作详情 | 因异常交易行为冻结 |
| status | 操作状态 | success / warning / error |
| approval_id | 关联审批ID（如有） | apr-20260618-0005 |

### 5.3 日志查询与分析

#### 5.3.1 查询维度

- **按操作人**：查看指定管理员的所有操作
- **按模块**：查看指定模块的操作记录
- **按时间**：按时间范围筛选
- **按IP地址**：追踪特定IP的操作
- **按操作类型**：筛选特定操作（如冻结、删除）
- **按状态**：成功/失败/警告

#### 5.3.2 分析功能

- **异常行为检测**：非常规时间登录、异常频率操作
- **操作关联性分析**：多账号协同操作检测
- **权限使用分析**：权限使用频率、未使用权限
- **合规性检查**：未审批操作、越权操作检测

### 5.4 日志安全

- **不可篡改**：日志写入后不可修改，关键日志上链存证
- **访问控制**：只有风控合规和超级管理员可查看审计日志
- **定期备份**：日志每日备份，异地存储
- **加密存储**：敏感字段加密存储

## 6. 安全防护机制

### 6.1 登录安全

#### 6.1.1 认证方式

| 安全层级 | 认证方式 | 适用角色 |
|---------|----------|----------|
| L1 | 账号密码 + 图形验证码 | 客服/数据分析师 |
| L2 | 账号密码 + 2FA (TOTP) | 业务管理员/财务 |
| L3 | 账号密码 + 2FA + IP白名单 | 运营/运维/风控总监 |
| L4 | 账号密码 + 2FA + IP白名单 + 硬件密钥 | 超级管理员 |

#### 6.1.2 登录安全策略

- **密码强度**：至少12位，包含大小写字母、数字、特殊字符
- **密码有效期**：90天强制更换，不可与最近5次重复
- **登录失败锁定**：连续5次失败锁定30分钟
- **会话超时**：30分钟无操作自动登出
- **异地登录提醒**：非常用IP登录需邮件/短信确认
- **设备绑定**：新设备登录需管理员审批

### 6.2 操作安全

#### 6.2.1 敏感操作二次确认

以下操作需要再次输入密码 + 2FA验证：
- 修改管理员密码
- 权限分配与回收
- 用户冻结/解冻
- 大额提现审核
- 系统配置变更
- 数据库高危操作

#### 6.2.2 操作窗口期

- **高危操作**：仅允许在工作时间（9:00-18:00）执行
- **系统维护**：仅允许在低峰期（02:00-06:00）执行
- **紧急操作**：需超级管理员特批，事后补流程

### 6.3 网络安全

#### 6.3.1 访问控制

- **VPN接入**：所有管理员必须通过企业VPN访问后台
- **IP白名单**：高权限账号限制指定IP段
- **多因素认证**：VPN登录也需MFA
- **网络隔离**：管理后台与业务系统网络隔离

#### 6.3.2 数据安全

- **传输加密**：全链路 HTTPS + TLS 1.3
- **存储加密**：敏感数据加密存储
- **数据脱敏**：查看时敏感信息脱敏显示
- **导出限制**：数据导出需审批，且加水印

### 6.4 账号安全

- **一人一号**：禁止账号共用
- **定期审计**：每月审计管理员账号清单
- **离职即销**：人员离职当日注销账号
- **休眠清理**：90天未登录账号自动禁用

## 9. 应急响应流程

### 9.1 应急事件分级

| 级别 | 定义 | 示例 |
|------|------|------|
| I级（特别重大） | 系统全面瘫痪/巨额资金风险/重大合规事件 | 数据库被删、交易所被黑 |
| II级（重大） | 核心功能故障/大额资金风险/严重安全漏洞 | 提现系统故障、用户数据泄露 |
| III级（较大） | 部分功能异常/一般安全事件 | 登录异常、个别功能不可用 |
| IV级（一般） | 轻微故障/小范围影响 | UI显示问题、非核心功能异常 |

### 9.2 应急响应组织

```
应急总指挥（超级管理员）
    ├── 技术组（运维+开发）：故障排查与修复
    ├── 业务组（运营+客服）：用户沟通与业务处理
    ├── 风控组（合规+安全）：风险评估与控制
    └── 外联组（公关+法务）：对外沟通与合规处理
```

### 9.3 应急响应流程

```
事件发现 → 初步判断 → 启动应急预案 → 成立应急小组
                                                  ↓
                                          事件调查与分析
                                                  ↓
                                          制定处置方案
                                                  ↓
                                          执行处置措施
                                                  ↓
                                          验证修复效果
                                                  ↓
                                          事件复盘与改进
```

### 9.4 典型应急预案

#### 9.4.1 系统被黑应急

1. **立即隔离**：切断可疑服务器网络连接
2. **保全证据**：保存日志、内存镜像等证据
3. **风险评估**：评估影响范围和损失
4. **系统修复**：清理后门、修复漏洞
5. **数据恢复**：从备份恢复数据
6. **用户通知**：通知受影响用户
7. **上报监管**：按规定上报监管部门
8. **事后复盘**：分析原因，加强防护

#### 9.4.2 数据库故障应急

1. **切换备库**：立即切换到备用数据库
2. **故障排查**：定位故障原因
3. **数据恢复**：如有数据丢失，从备份恢复
4. **验证数据**：核对数据一致性
5. **业务恢复**：逐步恢复业务
6. **根因分析**：找出根本原因，防止再发

#### 9.4.3 资金安全事件应急

1. **冻结相关账户**：立即冻结可疑账户
2. **追溯资金流向**：追踪资金去向
3. **冻结链上资产**：如涉及链上资产，联系相关方
4. **报警备案**：涉及盗窃立即报警
5. **用户沟通**：及时与用户沟通
6. **保险理赔**：如有保险，启动理赔

## 13. 自动化引擎协同机制

### 13.1 概述

管理员后台必须能够协同交易所的 **六大自动化引擎** 完成全部业务自动化工作。六大引擎分别为：

| 编号 | 引擎 | 核心能力 | 自动化场景 |
|------|------|---------|-----------|
| 26 | **AI分析中心** | 模型 / 知识库 / 风险预测 / 智能决策 | 用户风险评估、欺诈检测、智能决策支持、知识图谱查询 |
| 27 | **区块链** | 浏览器 / 节点 / 合约 / 存证链 | 上链存证、智能合约部署、链上查询、交易追溯 |
| 28 | **OpenClaw智能体** | 编排 / 市场 / 训练 / 监控 | 账户冻结、交易阻断、策略执行、邮件发送 |
| 29 | **n8n工作流** | 编辑器 / 模板 / 触发器 / 历史 | 通知发送、数据同步、Webhook调用、报表导出 |
| 30 | **AI大模型集成** | 模型管理 / Prompt / 推荐 / 成本分析 | 智能客服、报告生成、文案生成、Prompt工程 |
| 31 | **BPM工作流引擎** | 建模 / 运行 / 监控 / 分析 | 流程建模、审批流、流程编排、SLA监控 |

**协同原则**：

1. **统一调度**：所有引擎通过统一的 **自动化协同中枢（Automation Hub）** 调度，对管理员屏蔽底层差异。
2. **事件驱动**：引擎之间通过 **事件总线** 通信，事件触发工作流，工作流编排多引擎协同执行。
3. **可观测性**：所有引擎的运行状态、任务执行、事件流转都在协同中心统一展示。
4. **审批联动**：高风险的协同任务自动触发审批工作流，与管理员后台的审批体系深度集成。
5. **审计追溯**：每一次引擎协同执行都会自动记录到审计日志，关键操作上链存证。
6. **可编排性**：管理员可基于业务需求，自助编排多引擎协同工作流。

### 13.4 协同工作流

#### 13.4.1 工作流定义

协同工作流由多个 **步骤** 组成，每个步骤由特定引擎执行，支持：

- **依赖关系**：`dependsOn` 指定前置步骤
- **失败继续**：`continueOnError` 步骤失败不中断后续步骤
- **超时控制**：`timeout` 设置步骤超时时间
- **参数插值**：支持 `${event.payload.userId}` 语法引用上下文

#### 13.4.2 内置工作流模板

系统预置了 5 个核心工作流，覆盖主要业务场景：

| 工作流 | 触发事件 | 引擎协同链路 |
|--------|---------|-------------|
| **用户注册智能引导** | `user.registered` | AI分析中心（风险评估）→ AI大模型（生成欢迎语） |
| **大额提现风控审核** | `transaction.large.withdrawal` | OpenClaw（冻结账户）→ AI分析中心（风险分析）→ 区块链（存证）→ BPM（发起审批） |
| **可疑交易智能阻断** | `risk.alert` | AI分析中心（危害检测）→ OpenClaw（阻断交易）→ AI大模型（生成报告）→ 区块链（存证） |
| **代币上线全流程** | `token.listed` | AI分析中心（风险评分）→ BPM（审批）→ 区块链（部署合约）→ n8n（发送通知） |
| **系统配置变更审计** | `system.config.changed` | AI大模型（变更摘要）→ n8n（通知管理员）→ 区块链（存证） |

#### 13.4.3 自定义工作流

管理员可在协同工作面板中自定义工作流：

```typescript
automationHub.createWorkflow({
  name: 'KYC 智能审核',
  description: '用户提交KYC后由AI自动审核，存证到区块链',
  trigger: { type: 'event', eventType: 'user.kyc.submitted' },
  steps: [
    {
      id: 'step-1',
      engine: 'ai-center',
      action: 'risk_assessment',
      params: { userId: '${event.payload.userId}' },
    },
    {
      id: 'step-2',
      engine: 'openclaw',
      action: 'execute_strategy',
      params: { strategy: 'kyc_auto_approve', riskLevel: '${step-1.result.riskLevel}' },
      dependsOn: ['step-1'],
      continueOnError: true,
    },
    {
      id: 'step-3',
      engine: 'blockchain',
      action: 'evidence_store',
      params: { dataType: 'kyc_approval', data: '${step-1.result}' },
      dependsOn: ['step-1'],
    },
    {
      id: 'step-4',
      engine: 'n8n',
      action: 'send_notification',
      params: { channel: 'user', template: 'kyc_result' },
      dependsOn: ['step-2', 'step-3'],
    },
  ],
  enabled: true,
  approvalRequired: false,
  createdBy: 'admin-001',
});
```

### 13.6 与管理员后台运作机制的集成

#### 13.6.1 与 RBAC 权限的集成

- 工作流的执行权限受 RBAC 控制
- 引擎操作所需的权限点已注册到权限树
- 无权限的管理员无法触发或修改工作流

#### 13.6.2 与审批工作流的集成

当工作流配置 `approvalRequired: true` 时：

1. 工作流自动创建审批记录
2. 工作流暂停执行，等待审批结果
3. 审批通过后，工作流继续执行后续步骤
4. 审批驳回后，工作流终止并回滚

```typescript
// 工作流步骤中调用 BPM 引擎自动发起审批
{
  engine: 'bpm',
  action: 'start_approval',
  params: {
    type: 'large_withdrawal',
    data: { userId: '${event.payload.userId}', amount: '${event.payload.amount}' }
  }
}
```

#### 13.6.3 与审计日志的集成

- 每次工作流执行自动记录审计日志
- 每次引擎调用记录操作人、时间、参数、结果
- 关键操作（高风险、涉及资金）通过区块链引擎上链存证
- 审计日志可在管理员后台直接查询

#### 13.6.4 与告警机制的集成

- 引擎异常时自动触发告警
- 工作流执行失败率超过阈值时告警
- 引擎离线时立即告警
- 告警通过 n8n 工作流多渠道通知

### 13.9 安全与合规

#### 13.9.1 引擎调用安全

- 所有引擎调用都需要 JWT 鉴权
- 引擎调用通过 API 网关统一接入
- 引擎调用链全程记录到审计日志
- 关键操作（高风险）需要二次授权

#### 13.9.2 数据安全

- 跨引擎传输的数据统一加密（TLS 1.3+）
- 敏感数据在引擎适配层做脱敏处理
- 区块链存证的数据使用哈希存储
- AI 大模型调用不传输用户隐私信息

#### 13.9.3 合规要求

- 所有自动化操作必须可追溯
- 涉及资金的自动化必须经过审批
- 涉及 KYC/AML 的操作必须双重验证
- 涉及法币的操作必须符合当地法规

### 13.10 故障处理

#### 13.10.1 引擎故障

- **单引擎故障**：协同中枢自动重试，失败超过阈值后告警
- **引擎不可用**：相关工作流自动跳过该步骤并告警
- **引擎数据不一致**：自动回滚到上一个一致状态

#### 13.10.2 工作流故障

- **步骤失败**：根据 `continueOnError` 决定是否继续
- **工作流超时**：自动终止并告警
- **重试耗尽**：记录失败原因到审计日志，人工介入

#### 13.10.3 数据丢失

- 所有工作流执行都有幂等保证
- 关键状态持久化到数据库
- 区块链存证不可篡改，可作为最终一致性的依据
