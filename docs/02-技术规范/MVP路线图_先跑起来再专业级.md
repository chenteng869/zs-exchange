# ZS 交易所 · MVP 路线图（先跑起来 → 再专业级）

> **策略**: MVP 优先，跑起来再说，专业级后面迭代  
> **当前代码量**: ~21.2 万行  
> **MVP 目标代码量**: ~35 万行（新增 ~14 万行）  
> **专业级目标代码量**: ~134 万行（后面再做）

---

## 一、MVP 定义：什么叫"跑起来"？

### 1.1 MVP 核心标准

一个交易所 MVP，最起码要满足：

| # | 能力 | 说明 | 必须有？ |
|---|------|------|---------|
| 1 | **用户能注册登录** | 邮箱/手机号注册、登录、找回密码 | ✅ 必须 |
| 2 | **能看行情** | 交易对列表、K 线、深度、实时价格 | ✅ 必须 |
| 3 | **能下单交易** | 限价单、市价单、撤单、订单列表 | ✅ 必须 |
| 4 | **有撮合引擎** | 订单能真的成交，不是只存数据库 | ✅ 必须 |
| 5 | **能充币** | 生成充值地址、链上监听、到账确认 | ✅ 必须 |
| 6 | **能提币** | 提币申请、审核、链上广播、状态查询 | ✅ 必须 |
| 7 | **钱包余额** | 各币种余额、可用/冻结、流水记录 | ✅ 必须 |
| 8 | **管理员后台** | 用户管理、交易对管理、充提审核 | ✅ 必须 |

### 1.2 MVP **不做**什么（后面专业级再做）

| 模块 | MVP 阶段 | 专业级再做 |
|------|---------|-----------|
| **KYC/AML** | 简单的邮箱验证，不做实名 | Sumsub + Chainalysis + Travel Rule |
| **法币入金/出金** | ❌ 不做 | MoonPay/Transak + 银行卡 + 银行转账 |
| **合约/杠杆** | ❌ 不做 | 永续合约、交割合约、杠杆 |
| **Web3 钱包** | ❌ 不做 | 自托管钱包、DApp 浏览器、WalletConnect |
| **DID 身份** | ❌ 不做 | W3C DID、VC、零知识证明 |
| **OTC 场外** | ❌ 不做 | OTC 交易、商家系统 |
| **理财/质押** | ❌ 不做 | 活期、定期、Staking |
| **API 开放平台** | ❌ 不做 | 开放 API、文档、SDK |
| **APP 端** | ❌ 不做 | iOS/Android 原生 |
| **多语言** | 只做中文 | 中英日韩等 10+ 语言 |
| **高级风控** | 简单限额 | 实时风控引擎、反欺诈、行为分析 |
| **做市商系统** | ❌ 不做 | 做市策略、流动性管理 |

### 1.3 MVP 支持的链和币种

**先少后多原则**：

| 类别 | MVP | 专业级 |
|------|-----|--------|
| **公链** | ETH、BSC、TRON | 10+ 条链 |
| **币种** | BTC、ETH、USDT、USDC、BNB、TRX | 50+ 币种 |
| **交易对** | BTC/USDT、ETH/USDT、BNB/USDT、TRX/USDT | 100+ 交易对 |

## 三、MVP 缺口分析（还差什么？）

### 3.1 P0 级（必须有，不然不算 MVP）

| # | 功能 | 工作量 | 说明 | 优先级 |
|---|------|--------|------|--------|
| 1 | **撮合引擎跑起来** | 高 | 订单要能真的撮合成交，不能只存数据库 | 🔴 P0 |
| 2 | **真实链上充值监听** | 高 | 不能只生成地址，要真的监听链上交易，到账就加余额 | 🔴 P0 |
| 3 | **真实提币广播** | 高 | 提币申请要真的签名广播上链 | 🔴 P0 |
| 4 | **行情实时推送** | 中高 | K 线、深度、成交要实时更新，不能只靠刷新 | 🔴 P0 |
| 5 | **交易页联调** | 中 | 前端交易页要和后端 API 完全打通 | 🔴 P0 |
| 6 | **钱包页联调** | 中 | 充值、提币、余额、流水页面要能用 | 🔴 P0 |
| 7 | **管理员后台核心功能** | 中 | 用户管理、交易对管理、充提审核 | 🔴 P0 |

### 3.2 P1 级（有了更好，MVP 可以简化）

| # | 功能 | 工作量 | 说明 | 优先级 |
|---|------|--------|------|--------|
| 1 | **邮箱验证** | 低 | 注册后邮箱验证 | 🟡 P1 |
| 2 | **提现审核流** | 中 | 后台审核提币，自动/手动 | 🟡 P1 |
| 3 | **KYC 简化版** | 中 | 基础实名（可以先用假数据跑通流程） | 🟡 P1 |
| 4 | **充提币手续费** | 低 | 提币收手续费 | 🟡 P1 |
| 5 | **交易手续费** | 低 | 成交收手续费 | 🟡 P1 |
| 6 | **用户邀请** | 中 | 邀请码、返佣（可以先不加） | 🟡 P1 |

### 3.3 P2 级（MVP 不做，专业级再加）

所有 1.2 节里"不做"的都是 P2。

## 四、MVP 实施路线图

### 4.1 时间估算（AI 辅助，按你说的高强度模式）

**前提假设**：
- 1 人 + AI 深度辅助
- 每天有效工作 12-16 小时（不是 24 小时，人要睡觉）
- 目标是跑通核心流程，不是完美

| 阶段 | 时间 | 主要工作 | 新增代码量 |
|------|------|---------|-----------|
| **Phase 0：现状梳理** | 1 天 | 盘点现有代码、确认架构、理清楚哪些能用哪些要改 | 0 |
| **Phase 1：撮合引擎** | 3-5 天 | 把撮合引擎跑起来，订单能真的成交 | ~1.5 万行 |
| **Phase 2：钱包充提** | 5-7 天 | 链上监听、充值到账、提币广播、地址管理 | ~2 万行 |
| **Phase 3：行情实时化** | 2-3 天 | WebSocket 推送、K 线生成、深度更新 | ~1 万行 |
| **Phase 4：前端联调** | 3-5 天 | 交易页、钱包页、订单页全链路联调 | ~1.5 万行 |
| **Phase 5：管理后台** | 3-4 天 | 用户管理、交易对管理、充提审核能用 | ~1 万行 |
| **Phase 6：测试 & 修 Bug** | 3-5 天 | 全流程测试、修 Bug、优化体验 | ~1 万行 |
| **合计** | **20-30 天** | | **~8 万行** |

> 注意：上面的估算偏乐观，实际可能要乘以 1.5-2 倍的缓冲
> **保守估计：1.5-2 个月能跑通 MVP**

### 4.2 分阶段详细计划

#### Phase 1：撮合引擎跑起来（3-5 天）

**目标**：用户下单 → 撮合 → 成交 → 余额更新，全链路通

**具体工作**：
1. 梳理现有 matching 引擎代码，确认逻辑是否正确
2. 把撮合引擎集成到订单 API 里（现在下单只是存数据库，没进撮合）
3. 实现撮合结果持久化（成交记录、余额更新）
4. 初始流动性：用做市机器人或者直接接 Binance 价格当对手盘
5. 测试：挂单、撤单、成交、部分成交，都要测通

**难点**：
- 撮合引擎的并发和性能
- 余额冻结/解冻的正确性
- 部分成交的处理

---

#### Phase 2：钱包充提真能用（5-7 天）

**目标**：用户充币 → 链上到账 → 余额增加；用户提币 → 审核 → 上链 → 余额减少

**具体工作**：
1. 整理现有 wallet 模块，确认哪些能用
2. 接入真实 RPC 节点（Infura/Alchemy + TRON Grid）
3. 充值监听：监听链上交易，确认数够了就加余额
4. 地址生成：HD 钱包派生地址，用户充值地址唯一
5. 提币流程：申请 → （可选审核）→ 签名 → 广播 → 确认
6. 冷热钱包：先简化，只用一个热钱包就行
7. 支持 3 条链：ETH、BSC、TRON

**难点**：
- 链上交易确认数处理
- 私钥/助记词的安全存储（MVP 可以先存在环境变量里）
- 交易失败/回滚的处理

---

#### Phase 3：行情实时推送（2-3 天）

**目标**：交易页价格、K 线、深度、成交实时更新，不用手动刷新

**具体工作**：
1. 行情源：接 Binance 的 WebSocket 当行情源（MVP 先用别人的）
2. 实现我们自己的 WebSocket 推送
3. 交易页接入实时行情
4. K 线数据：从 Binance 拉历史，实时增量更新
5. 深度数据：实时维护订单簿快照

**难点**：
- WebSocket 连接管理和重连
- 数据推送的频率控制（不能太频繁把前端卡死）

---

#### Phase 4：前端联调（3-5 天）

**目标**：用户端所有核心页面都能用，流程走通

**具体工作**：
1. 交易页：下单、撤单、订单列表、成交记录，全联调
2. 钱包页：余额、充值地址、提币、流水，全联调
3. 会员中心：个人信息、安全设置（密码、TOTP）
4. H5 端同步适配
5. 错误处理、加载状态、空状态

**难点**：
- 前后端数据格式对齐
- 用户体验细节很多

---

#### Phase 5：管理后台核心功能（3-4 天）

**目标**：运营人员能通过后台管理用户和资产

**具体工作**：
1. 用户管理：列表、详情、禁用/启用
2. 交易对管理：上下架、调整参数
3. 充提审核：充值确认、提币审核
4. 数据概览：用户数、交易量、资产概览
5. 管理员权限：超级管理员 + 普通运营

**难点**：
- 权限控制要准确
- 敏感操作要有二次确认和日志

---

#### Phase 6：测试 & 修 Bug（3-5 天）

**目标**：核心流程稳定，没有致命 Bug

**具体工作**：
1. 全流程回归测试
2. 修发现的 Bug
3. 性能初步优化（数据库加索引、关键接口优化）
4. 安全初步检查（SQL 注入、XSS、越权访问）
5. 部署上线（测试环境 + 生产环境）

## 五、为什么这么快？（和 18 个月的区别）

### 5.1 MVP vs 专业级对比

| 维度 | MVP（1.5-2 个月） | 专业级（18 个月） |
|------|------------------|-----------------|
| **代码量** | ~35 万行 | ~134 万行 |
| **功能范围** | 8 个核心模块 | 20+ 个模块 |
| **支持链数** | 3 条 | 10+ 条 |
| **支持币种** | 5-10 个 | 50+ 个 |
| **KYC/AML** | 没有/简化 | Sumsub + Chainalysis + Travel Rule |
| **法币入金** | 没有 | MoonPay + 银行卡 + 银行转账 |
| **合约/杠杆** | 没有 | 永续 + 交割 + 期权 |
| **Web3 生态** | 没有 | 自托管钱包 + DApp 浏览器 + DID |
| **安全等级** | 基础 | HSM + 多签 + 渗透测试 + 审计 |
| **合规** | 不合规（先跑起来） | VARA 牌照 + 全面合规 |
| **性能要求** | 100 TPS 够了 | 10,000+ TPS |
| **可用性要求** | 99% | 99.99% |

### 5.2 核心差距：不是"快多少"，是"砍了多少"

MVP 能快，不是因为 AI 多厉害，而是因为**我们砍掉了 80% 的功能**：

```
专业级要做的：134 万行
MVP 要做的：  35 万行
砍掉的：      99 万行（74%）
```

砍的都是什么？
- 合规相关（KYC/AML/Travel Rule）：~10 万行
- 法币支付：~12 万行
- 合约交易：~5 万行
- Web3 钱包和 DApp 浏览器：~15 万行
- DID 身份：~6 万行
- 高级风控：~9 万行
- 机构/企业功能：~10 万行
- API 开放平台：~11 万行
- 运营后台增强：~10 万行
- 基础设施和测试：~15 万行
- ...

### 5.3 从 MVP 到专业级的路径

```
MVP（1.5-2 个月）
  ↓
增强版（+3-4 个月）：加 KYC、法币入金、风控
  ↓
专业版（+6-8 个月）：加合约、Web3 钱包、API 开放平台
  ↓
旗舰版（+6-8 个月）：全面合规、多链、机构级
```

合计约 16-22 个月，和之前说的 18 个月对得上。

## 六、风险与挑战

### 6.1 技术风险

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 撮合引擎有 Bug 导致资产不对 | 中 | 高 | 先在测试网跑，充分测试，有对账机制 |
| 钱包私钥管理不当被偷 | 中 | 极高 | MVP 用测试币，热钱包不要放太多钱 |
| 链上监听漏单/重复 | 中 | 中 | 幂等设计 + 定期对账补漏 |
| 性能不够卡 | 高 | 中 | MVP 用户少，问题不大，后面再优化 |

### 6.2 安全风险

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 被黑客攻击 | 中 | 极高 | 先不上主网，用测试网；代码审计后再上 |
| SQL 注入/XSS | 低 | 中 | Prisma 已经防 SQL 注入，前端做 XSS 过滤 |
| 越权访问 | 中 | 高 | 每个接口都做权限校验，写测试用例 |

### 6.3 MVP 阶段的安全底线

**MVP 阶段绝对不能做的事**：
1. ❌ 不要直接上主网，先用测试网
2. ❌ 热钱包不要放大量资金
3. ❌ 不要对外开放注册（仅限内部测试）
4. ❌ 不要接真实法币
5. ❌ 不要承诺任何资产安全

**记住：MVP 是用来跑通流程、验证想法的，不是用来做生意的。**

## 七、下一步行动

### 7.1 确认事项

在开始之前，需要确认几个问题：

1. **MVP 范围确认**：上面定义的 MVP 范围（注册/登录 + 行情 + 现货交易 + 充提币 + 管理后台）够吗？还是要加/减？

4. **行情源**：MVP 阶段直接用 Binance 的行情当数据源，没问题吧？
5. **撮合方式**：MVP 阶段是自己撮合，还是直接接 Binance 的价格做市？

### 7.2 如果确认了，就从这里开始

```
Day 1：梳理现有撮合引擎代码 + 设计集成方案
Day 2-3：撮合引擎集成 + 订单成交逻辑
Day 4-5：撮合联调测试 + 做市机器人（初始流动性）
...
```
