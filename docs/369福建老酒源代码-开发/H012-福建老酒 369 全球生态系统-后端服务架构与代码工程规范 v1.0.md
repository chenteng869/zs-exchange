# 6\. 分层代码规范



每个业务模块统一分层：



```Plain Text
Controller：接收 HTTP 请求
Service：业务逻辑
Repository：数据库访问
DTO：入参与出参
Entity：领域对象
Event：领域事件
StateMachine：状态机
Policy / Guard：权限与规则校验
```



禁止：



```Plain Text
Controller 里写业务逻辑
Service 里直接拼 SQL
Repository 里写业务判断
绕过 Service 直接改数据库
```



---



# 7\. Controller 规范



Controller 只负责：



```Plain Text
参数接收
参数校验
调用 Service
返回标准响应
```



示例：



```TypeScript
@Post()
@UseGuards(AuthGuard)
async createOrder(
  @CurrentUser() user: CurrentUserDto,
  @Body() dto: CreateOrderDto,
  @Headers('Idempotency-Key') idempotencyKey: string,
) {
  return this.orderService.createOrder(user.id, dto, idempotencyKey);
}
```



Controller 不允许：



```Plain Text
计算分账
发积分
写财务流水
写审计日志细节
直接操作 Prisma
```

# 8\. Service 规范



Service 负责业务流程。



示例：创建订单 Service 应做：



```Plain Text
1. 校验用户状态
2. 校验 KYC
3. 校验商品状态
4. 校验地区限制
5. 校验库存
6. 校验推荐关系
7. 调用风控预检查
8. 创建订单
9. 锁定库存
10. 发布 OrderCreated 事件
11. 写审计日志
```



Service 不应做：



```Plain Text
直接发送链上交易
直接改其他服务数据库
直接绕过资产账本改余额
```

# 9\. Repository 规范



Repository 只负责数据库读写。



示例：



```TypeScript
@Injectable()
export class OrdersRepository {
  constructor(private readonly prisma: PrismaService) {}

  async findById(orderId: string) {
    return this.prisma.orders.findUnique({
      where: { id: orderId },
    });
  }

  async create(data: Prisma.ordersCreateInput) {
    return this.prisma.orders.create({ data });
  }
}
```



Repository 不允许：



```Plain Text
判断订单能否支付
判断用户是否 KYC
计算奖励
调用其他服务
发事件
```

# 10\. DTO 规范



DTO 必须：



```Plain Text
使用 class-validator
字段明确
金额使用 string
枚举使用 enum
必填和可选清晰
```



示例：



```TypeScript
export class CreateOrderDto {
  @IsString()
  product_id: string;

  @IsInt()
  @Min(1)
  quantity: number;

  @IsString()
  currency: string;

  @IsOptional()
  @IsString()
  shipping_address_id?: string;

  @IsOptional()
  @IsString()
  referral_code?: string;
}
```



金额 DTO 示例：



```TypeScript
export class AmountDto {
  @IsString()
  amount: string;

  @IsString()
  currency: string;
}
```



禁止：



```Plain Text
金额使用 number
前端传浮点数直接计算
```

# 11\. Entity / Domain Model 规范



Entity 表示领域对象，不等于数据库表。



示例：



```TypeScript
export class OrderEntity {
  constructor(private readonly props: OrderProps) {}

  canPay(): boolean {
    return this.props.status === OrderStatus.PENDING_PAYMENT;
  }

  markPaid() {
    if (!this.canPay()) {
      throw new DomainException('ORDER_CANNOT_BE_PAID');
    }
    this.props.status = OrderStatus.PAID;
  }
}
```



目的：



```Plain Text
把状态判断和领域规则放进对象里
减少 Service 中散落 if else
```

# 12\. 状态机规范



所有核心对象必须使用状态机。



例如订单状态流转：



```Plain Text
created
→ pending_payment
→ payment_processing
→ paid
→ risk_checking
→ confirmed
→ fulfilling
→ fulfilled
→ completed
```



异常流转：



```Plain Text
pending_payment → cancelled
paid → refund_requested
refund_requested → refunded
paid → risk_hold
risk_hold → confirmed / cancelled
```



必须建立：



```TypeScript
order.state-machine.ts
```



示例结构：



```TypeScript
export const OrderStateTransitions = {
  created: ['pending_payment', 'cancelled'],
  pending_payment: ['payment_processing', 'cancelled'],
  payment_processing: ['paid', 'failed'],
  paid: ['risk_checking', 'refund_requested', 'risk_hold'],
  risk_checking: ['confirmed', 'risk_hold'],
  confirmed: ['fulfilling', 'refund_requested'],
  fulfilling: ['fulfilled'],
  fulfilled: ['completed'],
};
```



禁止：



```Plain Text
任意 update status
后台直接改状态
```

# 15\. 事务一致性规范



跨服务不能使用普通数据库事务。



必须采用：



```Plain Text
本地事务 + 事件驱动 + Outbox Pattern + 幂等消费
```



---



## 15\.1 Outbox Pattern



每个服务在本地事务中同时写：



```Plain Text
业务数据
outbox_events
```



然后由后台任务发送到 Kafka。



表：



```Plain Text
outbox_events
```



字段：



```Plain Text
id
event_type
payload
status
retry_count
next_retry_at
created_at
published_at
```



状态：



```Plain Text
pending
published
failed
dead
```



---



## 15\.2 为什么必须用 Outbox



避免这种问题：



```Plain Text
订单写入成功，但事件发送失败
事件发送成功，但订单写入失败
```



Outbox 可以保证：



```Plain Text
业务落库和事件记录在同一事务内
事件可重试
事件可追踪
```
