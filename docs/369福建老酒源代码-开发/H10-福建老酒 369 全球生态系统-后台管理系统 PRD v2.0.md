# 2\. 后台设计总原则



## 2\.1 高危操作必须审批



以下操作不得直接执行：



```Plain Text
后台加积分
后台扣积分
冻结资产
解冻资产
调整算力
调整释放额度
发放奖励
追回奖励
修改规则
修改分账比例
创建释放池
开放领取
财务结算
税务冲销
节点解封
NFT 撤销
金库转账
```



必须走：



```Plain Text
申请 → 审批 → 执行 → 审计
```



---



## 2\.2 所有操作必须留痕



后台所有操作必须记录：



```Plain Text
操作人
角色
动作
对象
修改前
修改后
原因
IP
设备
审批单
时间
```



审计日志：



```Plain Text
不可删除
不可篡改
可搜索
可导出
```



---



## 2\.3 只允许通过业务流程改数据



禁止后台直接改数据库。



例如：



```Plain Text
不能直接改用户积分余额
不能直接改订单金额
不能直接改奖励金额
不能直接改释放额度
不能直接删财务流水
不能直接删审计日志
```



正确做法：



```Plain Text
创建调整申请
审批通过
系统生成反向流水或调整流水
审计记录
```



---



## 2\.4 后台权限采用 RBAC \+ ABAC



RBAC：



```Plain Text
按角色授权。
```



ABAC：



```Plain Text
按地区、金额、业务类型、风险等级、节点等级、审批级别、时间范围授权。
```



例子：



```Plain Text
财务管理员只能处理自己权限范围内的结算。
风控管理员可以冻结资产，但解冻需要审批。
运营管理员可以创建活动，但不能改奖励比例。
客服可以查看订单，但不能处理财务。
```
