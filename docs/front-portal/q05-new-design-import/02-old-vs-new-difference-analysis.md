## 2. 原 H5 vs 新设计（移动端）

### 2.1 原 H5 现状

| 维度 | 原 H5 现状 |
|---|---|
| 入口 | `src/app/h5/page.tsx` + `src/app/h5/layout.tsx` |
| 子域数 | 17 大子域 |
| 路由数 | 110+ |
| 视觉风格 | 与桌面端一致（深蓝品牌色） |
| 关键模块 | 行情/交易/钱包/AI/DeFi/NFT/IDO/Game/Shop/Stake 等 |
| 状态 | 全部已上线，dev server port 3200 可访问 |

### 2.2 新设计预期变化

| 维度 | 新设计预期 | 差异 | 风险 |
|---|---|---|---|
| 视觉 | 全新移动端视觉 | 颠覆式 | 110+ 路由全部重写风险 |
| 底部 Tab | "H5 底部 Tab 是什么" | 需重新设计 | 路径规划变更 |
| 业务模块 | "哪些页面需要登录" | 重新划分 | 鉴权逻辑调整 |
| 公开访问 | "哪些页面公开访问" | 重新划分 | SEO 重新规划 |

### 2.3 路由差异建议

> **建议策略**：**视觉升级 + 保留路由 + 局部重构**。理由：
>
> 1. 110+ 路由全部重写**代价巨大**（估计 30-50 人天）
> 2. 现路由已有真实业务绑定（b1 wallet API 等）
> 3. 老用户收藏/书签/外部链接**不能失效**
> 4. 渐进式改造**风险可控**

## 3. 原 APP vs 新设计

> **重要发现**：仓库**无独立原生 APP 工程**（参考 00 §5）。

| 维度 | 原 APP 现状 |
|---|---|
| iOS 工程 | **不存在** |
| Android 工程 | **不存在** |
| React Native 工程 | **不存在** |
| Expo 工程 | **不存在** |
| 小程序工程 | **不存在** |
| 实际形态 | H5 + 桌面 Web 双端，**H5 可被 WebView 嵌入** |

### 新设计对 APP 的预期影响

- **情况 A**：Stitch 新设计仅是 Web 端（含 H5），**无需新增 APP 工程** → 当前 H5 视觉升级即可
- **情况 B**：Stitch 新设计要求**原生 APP 化** → 需新增独立 APP 工程（不在本任务范围）
- **情况 C**：Stitch 新设计要求**小程序化** → 需新增 `miniprogram/` 独立工程（不在本任务范围）

> **需用户明确**：Stitch 新设计对应的是 A / B / C 哪种形态？

## 8. 落地策略建议

### 8.1 推荐策略

1. **保持旧官网 `/`** 在线运行（不替换）
2. **新建预览路由** `/portal-preview` 或 `/front-preview` 承载新设计
3. **H5 视觉升级 + 保留路由**，仅在 `/h5` layout 套新视觉
4. **新增缺失的辅助页**（公告/帮助/风险/费率/隐私等）
5. **不删除旧文件**，所有新文件放 `src/app/front-portal/preview/**`
6. **后端能力未到位的页面统一打"数据接入中/内测中"标签**

### 8.2 灰度切换

1. P1：内部验收（开发+设计+运营）
2. P2：灰度切流（10% 流量到新首页）
3. P3：观察 24-72 小时
4. P4：100% 切换
5. P5：保留旧首页可回滚
