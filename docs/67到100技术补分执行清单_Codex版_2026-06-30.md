## 0. 总原则

### 0.1 从 67 到 100 的技术定义

当前 67 分说明项目已经有工程资产、数据库模型、API 主干、页面、钱包/Web3/DID 模块和基础健康检查。要到 100 分，必须把系统从“工程资产”推进到“可技术上线”的状态：

1. 交易闭环真实可跑。
2. 钱包充提真实可跑。
3. 资金流水可对账。
4. 密钥与签名不暴露明文私钥。
5. 后台接口有管理员权限边界。
6. 前端核心页面接真实 API。
7. 测试门禁可分层执行。
8. 生产运行态有监控、告警、回滚、健康检查。
9. 依赖漏洞、默认密钥、mock 数据等技术红线关闭。

### 0.2 Codex 执行规则

每次只让 Codex 做一个任务编号，例如 `T01`。不要一次丢 10 个任务，否则容易串线。

建议每个任务都按以下提示词开头：

```text
请执行 docs/67到100技术补分执行清单_Codex版_2026-06-30.md 中的 Txx。
要求：
1. 先阅读任务涉及文件和现有实现，不要重写架构。
2. 只改本任务范围内的文件。
3. 完成后运行任务指定验收命令。
4. 如果验收失败，继续修复直到通过，或明确写出阻塞原因。
5. 不处理合规/法务/商务内容，只处理技术工程。
```

### 0.3 通用验收门禁

除非单项任务另有说明，每个任务完成后至少运行：

```bash
npm.cmd run type-check:p0
npm.cmd run health:routes
```

涉及交易、钱包、余额、订单、行情的任务，额外运行：

```bash

```

涉及全局类型、DID、Web3、钱包、前端页面的任务，尽量运行：

```bash

```

涉及 perp 的任务运行：

```bash

```

## 2. A 域：钱包充提与资金闭环（+7 分）

### T01. 充值地址生成与链配置打通（+1）

**目标**：用户能够按链和币种获取稳定充值地址，地址状态可落库、可复用、可禁用。

**重点文件**：

- `src/app/api/v1/wallet/deposits/route.ts`
- `src/app/api/v1/wallet/address-book/route.ts`
- `src/lib/wallet-transfer/deposit-address.service.ts`
- `src/lib/wallet-transfer/repos/deposit-address.repo.ts`
- `prisma/schema.prisma`

**Codex 执行动作**：

1. 梳理现有充值地址 API、service、repo。
2. 确认 `WalletDepositAddress` 与用户、链、币种关系。
3. 实现或修复“按 userId + chain + currency 获取/创建地址”的幂等逻辑。
4. 地址不可重复分配给不同用户。
5. 补充错误返回：不支持链、不支持币种、地址生成失败。

**验收命令**：

```bash
npm.cmd run type-check:p0
npm.cmd run health:routes
```

**完成定义**：

- 同一用户重复请求同链同币种，返回同一 active 地址。
- 不同用户不会拿到同一充值地址。
- 地址记录可在数据库模型中追踪状态。

### T02. 真实充值监听状态机（+1）

**目标**：链上交易进入系统后，能按确认数推进状态：`detected -> confirming -> confirmed -> credited`。

**重点文件**：

- `src/lib/wallet-transfer/deposit-listener.service.ts`
- `src/lib/wallet/deposit-monitor.ts`
- `src/lib/wallet/deposit.ts`
- `src/lib/wallet-transfer/repos/chain-transaction.repo.ts`
- `src/lib/wallet-transfer/repos/deposit.repo.ts`
- `src/lib/wallet-transfer/repos/chain-cursor.repo.ts`

**Codex 执行动作**：

1. 梳理已有 listener / monitor。
2. 为监听结果建立统一事件结构：chain、txHash、from、to、amount、currency、blockNumber、confirmations。
3. 加入 cursor 机制，避免重复扫块。
4. 加入 txHash + logIndex / outputIndex 幂等键。
5. 支持确认数不足时不入账，只更新确认进度。

**验收命令**：

```bash
npm.cmd run type-check:p0
npm.cmd run health:routes
```

**完成定义**：

- 同一链上交易重复投递不会重复创建充值记录。
- 确认数不足不会提前加余额。
- listener 可通过 mock chain client 做本地测试。
