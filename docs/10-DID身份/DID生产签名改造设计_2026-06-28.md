# DID 生产签名改造设计（KMS/托管签名）

更新时间：2026-06-28
范围：Solana DID 创建、锚定、解析 API

---

## 1. 现状与风险

现有实现观察：

- DID 锚定接口当前仍接收调用方传入 privateKey。
- 生产环境已做保护，直接返回错误，提示必须走托管签名。
- 仓库脚本存在明文私钥样例，不能用于生产。

结论：

- 当前能力可用于开发与联调，不满足生产密钥治理要求。

---

## 2. 改造目标

- 目标 1：生产环境不接收、不存储、不日志打印私钥。
- 目标 2：DID 锚定统一通过签名适配层，支持 local 与 kms 双实现。
- 目标 3：所有签名请求具备 request_id 与幂等键，支持失败重试与审计追踪。
- 目标 4：链上失败时可重放，不重复锚定，状态一致。

---

## 3. 目标架构

调用链：

1. API 层接收锚定请求。
2. 生成 request_id 与 payload_hash。
3. 调用 SignerProvider.signTransaction。
4. 签名服务返回 signature 与 key_id。
5. 上链广播并记录 tx_hash。
6. 写入 user_did 与 did_anchor_jobs。

职责划分：

- API 层：鉴权、参数校验、幂等检查。
- 应用层：构建交易、重试策略、状态机推进。
- 签名层：统一抽象，屏蔽 local/kms 差异。
- 基础设施层：审计日志、监控、告警。

---

## 4. 新增接口契约

### 4.1 DID 锚定 API（建议）

请求：

POST /api/v1/did/solana/anchor

{
  "did": "did:sol:xxxx",
  "requestId": "uuid",
  "idempotencyKey": "did-anchor-did:sol:xxxx-v1",
  "simulate": false
}

响应：

{
  "success": true,
  "data": {
    "did": "did:sol:xxxx",
    "requestId": "uuid",
    "anchorStatus": "submitted",
    "txHash": "...",
    "explorerUrl": "..."
  }
}

约束：

- 不再接受 privateKey 字段。
- requestId 必填，用于全链路追踪。
- idempotencyKey 必填，用于去重。

### 4.2 签名服务抽象（内部）

interface SignerProvider {
  signTransaction(input: {
    requestId: string;
    keyRef: string;
    chain: "solana";
    payloadHash: string;
    unsignedTxBase64: string;
  }): Promise<{
    signatureBase64: string;
    keyId: string;
    algorithm: "ed25519";
    provider: "local" | "kms";
  }>;
}

说明：

- local 实现仅用于开发环境。
- kms 实现用于预生产与生产。

## 5. 数据模型建议

新增表 did_anchor_jobs：

- id
- request_id（唯一）
- idempotency_key（唯一）
- did
- key_ref
- payload_hash
- status（pending/submitted/confirmed/failed/replayed）
- tx_hash
- attempt_count
- last_error
- created_at
- updated_at

状态机：

- pending -> submitted -> confirmed
- pending/submitted -> failed -> replayed -> submitted

---

## 6. 重试与幂等策略

- 广播失败：指数退避，最多 3 次。
- 查询确认失败：轮询区块确认，超时进入 failed。
- 重放请求：同 idempotency_key 返回同一作业与结果，不重复广播。

建议阈值：

- 广播超时：10 秒
- 轮询间隔：3 秒
- 最大确认等待：120 秒

## 9. 验收标准

- 生产环境请求体包含 privateKey 时，直接拒绝并审计。
- 预生产连续 20 次 DID 锚定成功率 >= 99%。
- 重复请求（相同 idempotencyKey）不重复上链。
- 任意失败可通过重放恢复，最终状态一致。
